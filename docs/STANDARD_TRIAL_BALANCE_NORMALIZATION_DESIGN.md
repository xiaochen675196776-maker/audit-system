# 科目余额表标准化导入设计方案

## 1. 目标

新增“科目余额表标准化导入”能力：把客户原始科目余额表导入系统，经过字段映射、层级识别、客户科目到全局标准科目映射、数据清洗、金额拆分和校验后，生成系统标准科目余额表。

第一版只完整实现“科目余额表”。“序时账”和“辅助明细账”先在数据查看入口中占位，后续单独设计。

## 2. 核心结论

- 使用现有 `/data/import` 导入页扩展向导，不新开完全独立的导入入口。
- 新导入名称为：`科目余额表标准化导入`。
- 使用系统内置的 `科目余额表.xlsx` 作为全局标准科目模板；该模板不是客户导入材料。
- 全局标准科目模板是系统主数据的唯一权威来源，只能由系统初始化或维护流程更新。
- 内置模板更新时采用全量同步：模板中没有的旧标准科目自动停用，不物理删除。
- 标准余额表历史数据保留导入当时的标准科目快照。
- 客户原始科目代码可选，科目名称建议必需；没有名称但有代码也允许进入待映射。
- 客户原始表不要求有余额方向、科目类别。
- 未映射到启用标准科目的客户科目阻止最终入库。
- 客户科目映射经验采用“客户优先 + 全局候选”。
- 已停用标准科目上的历史映射不自动套用，只作为警告候选。
- 标准余额表只保存最末级真实金额行。
- 父级金额不重复入库，只用于展示汇总和差异警告。
- 保留客户原始导入行快照，包括父级和末级。
- 科目余额表标准化导入必须是一个连续向导：字段确认后进入层级识别和科目匹配确认，不能直接导入完成。
- `/data/view` 是导入完成后的数据查看页，不是字段确认后的下一步，也不能替代科目确认步骤。
- 用户在科目匹配页手动选择启用标准科目后，`未映射`、`按标准方向拆分但暂缺方向` 这类可解决阻止项必须按当前确认映射动态重新计算，不能继续沿用 `analyze` 阶段的旧错误。

## 3. 标准科目模板

当前标准模板源文件来自：

```text
C:\Users\陈锐\Desktop\科目余额表.xlsx
```

该文件用于生成系统内置标准科目主数据。运行时不能依赖用户桌面路径，必须把模板内容作为系统资源或种子数据随项目发布。普通用户和客户导入流程不得上传、替换或维护这张标准模板。

新版模板结构：

| 列 | 名称 | 用途 |
| --- | --- | --- |
| A | 科目代码 | 标准科目代码 |
| B | 科目名称 | 标准科目名称 |
| C | 余额方向 | 标准余额方向，支持 `借` / `贷`，可为空 |
| D | 科目类别 | 标准科目类别，自由文本，可为空 |
| E-J | 期初/本期/期末借贷金额 | 作为系统标准余额字段说明，不作为标准科目表字段导入 |

标准科目内置初始化/维护规则：

- `科目代码` 和 `科目名称` 是标准模板自身的关键字段。
- `余额方向` 和 `科目类别` 可为空；为空不阻止标准科目模板导入。
- 标准方向为空时，如果客户金额列选择“按标准方向拆分”，对应客户行需要用户改成显式借方或贷方，否则不能入库。
- 同一 Excel 内重复科目代码阻止标准模板导入。
- 系统维护人员更新内置模板后重新同步时：
  - Excel 中已有代码且系统中存在：更新名称、方向、类别、级次、父级。
  - Excel 中有而系统中没有：新增。
  - 系统中有而 Excel 中没有：`is_active=false`，不删除。
- 第一版不提供普通用户前端上传标准模板的入口。

## 4. 客户原始科目余额表规则

客户表字段容忍度：

- 客户科目代码：可选。
- 客户科目名称：建议必需；没有名称但有代码也可进入待映射。
- 余额方向：可选。
- 科目类别：可选。
- 年度/期间：可由模板默认值或用户手填补齐。
- 金额列：不要求六列齐全，但至少要有一个可映射的期初余额、本期发生额或期末余额/发生额列。

阻止入库的错误：

- 某个参与入库的末级客户科目无法映射到启用标准科目。
- 参与入库行既没有客户科目代码，也没有客户科目名称。
- 参与入库行金额无法解析为数字。
- 年度/期间缺失且用户未补齐。
- 金额列选择“按标准方向拆分”，但标准科目方向为空且用户未改为显式借/贷。

可确认继续的警告：

- 父级原始金额与子级末级汇总金额不一致。
- 单列金额按方向拆分时出现负数反向余额。
- 历史映射指向已停用标准科目。
- 全局候选名称相似但非完全匹配。
- 同一客户科目存在多个候选标准科目。

## 5. 金额映射与拆分

第一版支持两种金额模式。

### 5.1 借贷两列直接映射

客户表已有：

```text
期初借方余额 / 期初贷方余额
本期借方发生额 / 本期贷方发生额
期末借方余额 / 期末贷方余额
```

用户在字段映射步骤直接映射到标准字段。

### 5.2 单列金额按方向拆分

客户表只有：

```text
期初余额
本期发生额
期末余额
```

用户可以选择该金额列的拆分方式：

- 直接进借方列。
- 直接进贷方列。
- 按标准科目方向拆分。

拆分规则：

```text
显式借方：
  正数进 debit
  负数取绝对值进 credit，并生成反向余额警告

显式贷方：
  正数进 credit
  负数取绝对值进 debit，并生成反向余额警告

按标准方向：
  标准方向=借：正数进 debit，负数进 credit
  标准方向=贷：正数进 credit，负数进 debit
  标准方向为空：阻止该行入库，除非用户改为显式借/贷
```

本期发生额也按同一规则处理。默认建议按标准科目方向拆分，但用户可在确认页修改。

## 6. 科目层级识别

### 6.1 有客户科目代码

按代码前缀自动推断层级和父级：

- 对每个客户科目代码，寻找已存在的最长上级代码前缀。
- 找到上级则设置 `parent_key`。
- 没有上级则为一级节点。
- 有子级的行为父级汇总行。
- 没有子级的行为末级明细行。

### 6.2 无客户科目代码但有 Excel 缩进

按 Excel 缩进级别给层级建议：

- 缩进越深，级次越低。
- 系统展示识别结果，用户确认。
- 用户可以调整层级、标记父级/末级、或一键改为平铺。

### 6.3 无代码且无缩进

默认平铺：

- 每行都作为末级明细候选。
- 用户可手动标记层级。

### 6.4 入库与展示

- 标准余额表只保存末级、参与入库的真实金额行。
- 父级行保留在原始行快照中，不进入标准余额表金额。
- 页面展示时按客户科目树动态汇总，默认展示父级，展开后显示下一级。
- 父级原始金额与子级汇总金额不一致时生成警告，由用户决定是否继续。

## 7. 客户科目到标准科目映射

映射推荐优先级：

1. 同一客户历史确认：`company_id + client_account_code + client_account_name`。
2. 同一客户名称匹配：`company_id + client_account_name`。
3. 标准科目代码相等：客户科目代码等于标准科目代码。
4. 全局历史确认：`client_account_code + client_account_name`，只作为候选。
5. 全局名称相似：客户科目名称与标准科目名称相似，只作为候选。

自动套用规则：

- 同一客户历史确认且标准科目仍启用：可高置信自动推荐。
- 全局候选不静默确认，需要用户确认。
- 历史映射指向停用标准科目：不自动套用，只作为警告候选。

保存经验：

- 用户确认客户科目映射后保存到 `client_account_mappings`。
- 同一客户同一客户科目再次确认同一标准科目时累计使用次数。
- 用户改选标准科目时，旧 active 经验停用，新经验启用。
- 全局候选第一版可以由所有客户确认数据汇总生成，但不能覆盖客户级经验。

## 9. API 设计

### 标准科目主数据

- `GET /api/v1/standard-accounts`
- `GET /api/v1/standard-accounts/{id}`

公开 API 只提供查询能力。标准科目由系统内置种子数据初始化，不属于客户导入流程。

内部维护流程需要支持：

- 从随项目发布的内置模板资源加载标准科目。
- 首次初始化时写入标准科目表。
- 模板版本更新时按全量同步规则更新、停用、恢复标准科目。
- 返回新增、更新、停用、错误行、警告行，供维护日志和测试使用。

如果保留 `POST /api/v1/standard-accounts/import`，它只能作为内部维护接口，不能在普通产品前端暴露；在没有权限体系前，优先改为不注册公开路由或仅保留测试/脚本级服务函数。

### 科目余额表标准化导入

建议新增独立后端路由，但前端入口仍在现有 `/data/import`：

- `POST /api/v1/standard-trial-balance-imports/preview`
- `POST /api/v1/standard-trial-balance-imports/{batch_id}/analyze`
- `POST /api/v1/standard-trial-balance-imports/{batch_id}/execute`
- `GET /api/v1/standard-trial-balance-imports/{batch_id}`

`preview`：

- 上传客户原始文件。
- 返回列 ID、表头、样例行、模板候选、字段建议。
- 生成 draft batch 或 file token。

`analyze`：

- 输入字段映射、金额映射、年度、期间。
- 提取客户科目清单。
- 识别层级。
- 返回客户科目树、标准科目候选、错误和警告。
- 这是字段确认后的唯一下一步。前端完成字段确认后必须调用 `analyze`，并进入层级/科目匹配确认页。
- `analyze` 不允许生成标准余额表，也不允许标记导入完成。

`execute`：

- 输入最终客户科目映射、层级确认、警告确认标志。
- 校验无未映射末级科目。
- 写入 raw rows 和 standard entries。
- 保存客户科目映射经验。
- 只有用户在科目匹配页确认全部末级科目映射，并处理阻止项/警告项后，前端才允许调用 `execute`。

### 数据查看

- `GET /api/v1/standard-trial-balances/tree`
- `GET /api/v1/standard-trial-balances/batches`

筛选：

- `company_id`
- `fiscal_year`
- `period`
- `batch_id`
- `view_mode=standard|client`

## 10. 前端设计

### 10.1 导入页

现有 `/data/import` 新增数据类型：

```text
科目余额表标准化导入
```

该类型使用扩展向导：

1. 上传文件。
2. 字段映射。
3. 层级识别。
4. 科目映射。
5. 清洗校验。
6. 入库完成。

强制流程规则：

- 字段映射确认按钮文案应表达为“下一步：识别科目和层级”，不能表达为“执行导入”或“完成导入”。
- 字段映射确认后调用 `POST /standard-trial-balance-imports/{batch_id}/analyze`。
- `analyze` 成功后必须展示客户科目树、层级识别结果、标准科目候选、未映射阻止项和警告。
- 用户完成科目匹配和警告确认前，不得调用 `execute`。
- 最终 `execute` 成功后，才显示导入完成，并可跳转 `/data/view`。
- 标准化导入可以作为 `/data/import` 内的导入类型出现，但不能退化为原有三步导入的“字段映射后直接执行”流程，也不能把科目确认拆成另一个独立菜单让用户自己再去找。

字段映射需要支持：

- 客户科目代码。
- 客户科目名称。
- 年度。
- 期间。
- 余额方向。
- 期初/本期/期末金额列。
- 金额列拆分方式：借方、贷方、按标准方向。

科目映射页需要支持：

- 客户科目树。
- 标准科目候选。
- 全部未映射项筛选。
- 停用标准科目警告候选。
- 用户手动选择标准科目。
- 一键确认高置信客户历史映射。

校验页需要支持：

- 错误列表：阻止入库。
- 警告列表：用户确认后可继续。
- 父级金额与子级汇总差异。
- 负数反向余额。

### 10.2 标准科目表查看

建议新增页面：

```text
/data/standard-accounts
```

功能：

- 展示系统内置标准科目表。
- 显示标准模板来源说明：系统内置，不由客户上传。
- 可筛选启用状态、科目类别、余额方向。
- 不提供上传、替换、全量同步按钮。
- 若标准科目未初始化，提示联系系统维护人员，不引导普通用户上传模板。

### 10.3 数据查看

新增页面：

```text
/data/view
```

三个 tab：

- 科目余额表：第一版可用。
- 序时账：占位，提示后续接入。
- 辅助明细账：占位，提示后续接入。

科目余额表默认按标准科目口径展示：

- 被审计单位。
- 年度。
- 期间。
- 导入批次。
- 显示口径：标准科目 / 客户原始科目。
- 树形展开：默认父级，展开下一级。
- 金额列：期初借/贷、本期借/贷、期末借/贷。

## 12. 第一版不做

- 不做序时账标准化导入。
- 不做辅助明细账标准化导入。
- 不做标准科目版本管理。
- 不物理删除标准科目。
- 不自动创建临时标准科目。
- 不允许普通用户上传或替换系统标准科目模板。
- 不用 LLM 自动判断科目映射。

- 不在标准余额表中保存父级汇总金额行。
