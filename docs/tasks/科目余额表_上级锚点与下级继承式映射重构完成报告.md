## 2. 架构改造

### 2.1 原流程（旧）

```text
每一个客户科目 → 独立匹配标准科目 → 产生多个候选
→ 每一个末级科目都要自动确认或人工确认
```

### 2.2 新流程

```text
先解析完整客户科目树 → 识别结构汇总节点 → 发现首批映射锚点
→ 只对锚点和继承中断点执行完整标准科目推荐
→ 沿树向下传播 + 检查继承中断规则
→ 普通下级默认继承最近锚点（不参与独立匹配）
→ 用户可将继承行单独映射为显式覆盖
→ 只有末级参与行生成标准余额表 entry
→ execute 只提交锚点 / 中断点 / 显式覆盖
```

## 10. 最终结论

✅ **重构完成**。本次重构将「每个末级客户科目独立匹配标准科目」的主流程改为「上级锚点 + 下级继承式映射」的主流程，显著减少了用户需要逐行确认的工作量；保留客户原始科目树不要求扩充标准科目库；研发费用化/资本化、原值/备抵、应收/应付等会计性质变化时正确触发继承中断；execute 阶段严格校验唯一性，移除了 `candidates[0]` 兜底；映射经验仅保存有价值的锚点和显式覆盖。
