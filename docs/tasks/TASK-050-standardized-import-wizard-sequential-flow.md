## 背景

总指挥复核发现：当前科目余额表标准化导入在字段确认后，表现为导入完成或进入原有导入结束流程，没有继续进入“层级识别、科目匹配、警告确认”等下一步。

这不符合既定设计。科目余额表标准化导入必须是一个连续向导：

```text
上传客户原始科目余额表
→ 字段映射与金额列确认
→ 层级识别
→ 客户科目匹配标准科目
→ 阻止项/警告项确认
→ 执行入库
→ 数据查看
```

字段确认后只能进入 `analyze` 和科目确认步骤，不能直接 `execute`，也不能显示“导入完成”。

## 目标

修正 `/data/import` 中“科目余额表标准化导入”的前端状态机和交互流程，让字段确认后自动进入科目确认链路，直到全部末级科目映射完成后才允许最终入库。

## 必须修正

1. 标准化导入不能走原有三步导入的最终执行逻辑。
   - 选择 `科目余额表标准化导入` 后，字段确认按钮不得调用旧 `/imports/execute`。
   - 字段确认后不得显示“导入完成”。
2. 字段确认后的下一步必须是 `analyze`。
   - 调用 `POST /api/v1/standard-trial-balance-imports/{batch_id}/analyze`。
   - 请求必须带字段映射、金额列拆分方式、年度、期间、层级识别模式。
3. `analyze` 成功后必须进入科目确认页面。
   - 展示客户科目树。
   - 展示层级来源：代码、缩进、平铺、手动。
   - 展示父级/末级标识。
   - 展示每个末级客户科目的标准科目候选。
   - 展示停用标准科目候选为警告候选，不自动套用。
4. 未完成科目确认时禁止最终入库。
   - 任何末级行未映射到启用标准科目时，`执行入库` 按钮禁用。
   - 禁用原因必须在页面中文展示。
5. 警告项必须在最终入库前展示。
   - 父级金额与子级汇总不一致。
   - 负数金额。
   - 缩进推断层级需要确认。
   - 全局候选或停用候选需要用户确认。
6. 最终 `execute` 只能在以下条件满足后调用：
   - 全部末级科目已映射到启用标准科目。
   - 阻止项为 0。
   - 需要确认的 warning 已由用户确认。
7. `execute` 成功后才显示导入完成。
   - 显示标准余额表条目数。
   - 显示映射经验保存数量。
   - 提供进入 `/data/view` 的按钮。
8. 不要把“科目匹配”做成用户要另找的独立功能入口。
   - 可以保留 `/data/view` 作为导入完成后的查看页。
   - 但 `/data/view` 不能承担科目确认职责。

## 建议前端步骤

标准化导入在 `/data/import` 内建议使用 5 个明确步骤：

1. `上传文件`
2. `字段与金额确认`
3. `科目层级与匹配`
4. `校验与确认`
5. `入库完成`

如果现有 UI 为 4 步，也可以合并第 3/4 步，但必须满足：字段确认后进入科目确认，最终入库前有阻止项和警告项确认。

### 完成内容

#### 1. 向导步骤重构（4→5 步）

标准化导入从原来的 4 步改为 5 步连续向导：

| 步骤 | 名称 | 功能 |
|------|------|------|
| 1 | 上传文件 | 上传客户科目余额表，填写客户标识、年度、期间 |
| 2 | 字段与金额映射 | 列映射、金额拆分方式配置、层级识别模式 → 调用 `analyze` |
| 3 | 层级与科目匹配 | 查看客户科目树、层级来源、候选映射、手动选择标准科目 |
| 4 | **校验与确认（新增）** | 展示阻止项/警告项/未映射科目，要求显式确认警告 |
| 5 | 入库完成 | 执行导入 → 显示结果（条目数、映射经验数）+ 进入 `/data/view` |

#### 2. API 调用链

- **字段确认**：调用 `POST /api/v1/standard-trial-balance-imports/{batch_id}/analyze`（传入字段映射、金额配置、年度、期间、层级模式）
- **科目匹配**：步骤 2 中 `analyze` 返回的 `mapping_recommendations` 展示候选；用户通过候选列表或搜索手动选择
- **执行入库**：仅在步骤 4 中，所有条件满足后调用 `POST /api/v1/standard-trial-balance-imports/{batch_id}/execute`（传入 `confirmed_mappings`、`warnings_confirmed`、`save_mapping_experience`）

#### 3. 未映射阻止项展示

- 步骤 2 底部：`stdConfirmHint` 提示未映射科目数量，按钮禁用直到全部映射
- 步骤 4 顶部：红色 `el-alert` 显示阻止入库错误 + 错误分类标签
- 步骤 4 中段：红色警告块显示未映射科目数量，要求返回上一步
- 按钮禁用时有 `footer-hint` 中文说明具体原因

#### 4. 最终 execute 启用条件

`stdCanExecute` 计算属性要求三个条件全部满足：
1. `stdBlockingErrors.length === 0` — 无阻止入库的错误
2. `stdUnmappedCount === 0` — 所有末级科目已映射
3. 如果 `stdWarnings.length > 0`，则必须 `stdWarningsConfirmed === true` — 警告已勾选确认

不满足时 `stdExecuteHint` 显示具体原因（"还有 X 条错误" / "还有 X 个科目未映射" / "请先勾选确认以上 X 条警告"）

#### 5. 警告确认机制

- 步骤 4 新增 `el-checkbox`："我已确认以上 N 条警告，了解可能的数据差异风险，仍然继续入库"
- `warnings_confirmed` 值从硬编码 `true` 改为使用实际的 `stdWarningsConfirmed.value`
- 切换映射后自动重置警告确认状态（用户需重新评估）
- 重新进入步骤 3（analyze 后）或从步骤 2 向前（stdGoConfirm）时重置确认状态

#### 6. 映射摘要

步骤 4 新增映射摘要表格，显示：
- 客户科目代码/名称 → 标准科目代码/名称
- 映射来源（客户历史/全局历史/代码匹配/名称相似/手动选择）
- 警告候选标识
