## 3. TASK-089 §1–§10 修复明细

### §1. 语义组匹配后再验证备抵语义一致性
**问题**：客户名称命中语义组（如 `fixed_assets`、`intangible_assets`）时，若标准科目是备抵/累计类（1702 累计摊销 / 1602 累计折旧 / 1471 跌价准备等），但客户名/路径都不含备抵语义，旧的语义组硬编码返回 `compatible` 是错的。

**修复**：`evaluate_name_compatibility` 中命中语义组后立即检查：
```python
sa_is_reserve = _has_negative_reserve_semantics(sa_name)
effective_client_is_reserve = client_is_reserve or path_is_reserve
if sa_is_reserve and not effective_client_is_reserve:
    return Conflict(...)  # 标准是备抵类,客户无减值语义
if effective_client_is_reserve and not sa_is_reserve:
    return Conflict(...)  # 客户含减值语义,标准是原值类
```

**典型防错案例**：「管理费用/无形资产摊销」不再误匹配「1702 减：无形资产-累计摊销」。

---

### §2. 其他应付/保证金不再误配其他应收款
**问题**：「保证金」作为叶子名不应单独决定资产/负债方向。当路径或父级明确为「其他应付款」时，必须保持负债方向。

**修复**：调整 `_SEMANTIC_ACCOUNT_GROUPS["other_receivables"]`：
- `client_aliases` 删除「保证金」（避免单 token 误判）
- `negative_aliases` 增加「其他应付款」（双向防护）

**典型防错案例**：「其他应付款/外部单位/保证金」 → 2241 其他应付款（不再误匹配 122101 其他应收款）。

---

### §3. 多 token 锚点匹配
**问题**：客户「周转材料__包装物」应既能匹配「周转材料」父级也能匹配「包装物」子级。原逻辑只看第一个 token。

**修复**：
- `_check_code_match_name_conflict` 加入多 token 候选锚点检查
- `_build_code_match_candidate` 的锚点匹配扩展到所有 token
- `_client_name_starts_with_standard_name` 任一 token 命中即视为命中
- 配套在 §5.3 增加单 token 全名的安全兜底（防止「营业收入」被误判为冲突）

---

### §4. 精确代码匹配永不降级
**问题**：旧逻辑中即使 client_code 完全等于 sa_code（精确命中），name_prefix 仍会追加与之冲突（指向不同 sa）的兜底候选，导致歧义。

**修复**：
- `_add_fallback_candidate`：若 entry 已有精确代码命中（`client_code == sa_code`），跳过与之冲突的兜底
- `_resolve_exact_code_vs_exact_name_conflict`：精确代码命中不参与降级，保护最强信号
- `_resolve_exact_code_vs_exact_name_conflict` 中追加 `name_prefix` 时，若已有精确代码命中则直接返回

---

### §5. 安全候选三重防护
#### §5.1 空 standard_account_id 一律视为不安全
新增 `_is_safe_candidate` 防护：候选缺少 `standard_account_id` 一律不视为安全候选，杜绝脏候选被自动确认。

#### §5.2 唯一安全候选判定
维持 `pick_unique_auto_confirm_candidate` 不变——多个不同安全目标时不自动确认，保留人工确认。

#### §5.3 废弃函数不得回退首项
`_pick_auto_confirm_candidate` 改为薄包装，仅转发到 `pick_unique_auto_confirm_candidate`；不再回退到首项候选。

### §7. 重大错配检测器升级
**问题**：TASK-088 的检测器过度敏感（8 条误报）和方向盲区（无法识别路径证据）。

**修复**：
- 仅在 picked 为安全自动确认且 `compatibility_status == "compatible"` 时检查（避免与 conflict 重复报警）
- 路径作为方向证据：`direction_text = client_name + path`
- 资产/负债方向：路径含「其他应付款」时优先于叶子「保证金」
- 原值/备抵方向：路径含「管理费用/销售费用/制造费用」时也视为原值方向
- 成本/收入：基于明确成本语义组（排除「摊余成本」/「成本价」/「历史成本」）

#### 防错案例覆盖
| 案例 | 旧检测器 | 新检测器 |
|---|---|---|
| 其他应付款/外部单位/保证金 → 其他应收款 | 漏报 | 命中负债→资产 |
| 管理费用/无形资产摊销 → 累计摊销 | 漏报 | 命中原值→备抵 |
| 投资收益__以摊余成本计量的金融资产终止确认收益 → 投资收益 | 误报为成本→收入 | 正确放行 |
| 其他存货 → 存货跌价准备 | 漏报 | 命中原值→备抵 |
| 固定资产减值准备 → 固定资产原值 | 漏报 | 命中备抵→原值 |

---

### §8. 包装物_纸箱 → 141101 包装物优先于 1411 周转材料
**问题**：客户「1411 包装物_纸箱」代码匹配 1411 周转材料，但名称首段「包装物」是更精确的子级 141101 包装物。旧逻辑会让父级独占自动确认。

**修复**：通过 §3 的多 token 锚点检查 + 现有 name_prefix 路径，让 1411 周转材料被降级为冲突候选，141101 包装物独占自动确认。

**回归测试**：`TestPackagingBoxNotMatchedToTurnoverParent::test_packaging_box_prioritizes_packaging_detail` ✅
